=== REQUIREMENTS DOC ===
1) Product Goal
Detect and explain when the same business KPI/metric is:
calculated in multiple SSRS reports (duplication), and/or
calculated differently (metric drift), and/or
recalculated in BI instead of Snowflake (layer violation),
and produce actionable rationalization recommendations (standardize, consolidate, move logic to Snowflake, retire report).
2) Scope & Inputs
In-Scope Inputs
A. SSRS artifacts
Report catalog (RDL files) OR access to ReportServer database
Embedded datasets / shared datasets
Data sources & connection strings (redacted acceptable)
Report parameters and default values
Subreports and drillthrough links (optional but valuable)
B. Snowflake
Database metadata: schemas, tables, views, columns, tags
View definitions / stored procedures (where permitted)
Query history (optional) for usage and runtime profiling
Roles/permissions read-only
Out-of-Scope
CRD/Alpha operational logic (unless later integrated)
Real-time streaming / trading workflows
3) Core User Personas (who uses the accelerator)
BI Lead / Data Architect – wants KPI truth and semantic alignment
Reporting Modernization Lead – wants which reports to consolidate/retire
Business Owner (Ops/Finance/PMO) – wants confidence in “one number”
Compliance / Audit Stakeholders – want lineage + definition consistency
4) Functional Requirements
FR1 — SSRS Intake & Parsing
Ingest SSRS reports from:
RDL files (bulk upload), and/or
ReportServer DB extraction
Extract:
Dataset queries (SQL text)
Data sources (server/db/schema)
Parameters + prompts + defaults
Expressions and calculated fields (if present)
Report structure: tables, matrices, charts (optional)
Output a normalized “Report Spec” JSON per report:
report_id, name, folder, datasets[], parameters[], dependencies[]
Acceptance: 95%+ of reports parsed without manual intervention; remainder flagged with parse errors.
FR2 — SQL Canonicalization & AST Generation
For every extracted query:
Parse SQL to an AST (or intermediate representation)
Normalize:
whitespace/comments
aliases
join ordering (where safe)
parameter substitution tokens
date filters normalization (EOD, T+1 patterns)
Store:
canonical_sql
sql_ast
referenced_objects (tables/views/columns)
Acceptance: same SQL with different formatting yields identical canonical signatures.
FR3 — Metric/KPI Extraction (Semantic)
Detect metrics in queries and expressions:
Aggregations (SUM, AVG, COUNT, etc.)
Derived measures (Return, P&L, Exposure, NAV, TE, etc.)
Calculated expressions (case when, window functions)
Build a Metric Definition Object:
metric_name (inferred)
expression_signature (hash of normalized AST sub-tree)
grain (portfolio/fund/security/date)
filters applied
join path signature
source tables/views
parameters impacting metric
Acceptance: for a test set of known reports, extracted metrics match human labeling at >80% precision, >70% recall (tunable).
FR4 — Metric Similarity & Clustering
Identify metrics that are “the same” across reports using:
expression signature similarity (AST-based)
semantic similarity (LLM embeddings on expression + column names)
context similarity (grain + joins + filters)
Output clusters:
Cluster ID
Member metric instances (report, dataset, field)
Confidence score
Acceptance: clusters group at least 80% of true duplicates together on sample validation set.
FR5 — Drift Detection & Drift Explanation
For each cluster, detect drift types:
Formula Drift
Same metric name/intention, different math
Filter Drift
different date logic (trade date vs settle date), cash inclusion, region filters, etc.
Join Drift
different source table/view or join paths leading to different populations
Aggregation Drift
SUM vs AVG, granularity mismatch, windowing differences
Generate a human-readable explanation:
“Metric X differs because Report A uses settle_date while Report B uses trade_date”
“Report C excludes CASH instruments; others include”
Acceptance: drift explanations include at least one concrete difference (filter/join/formula) for >90% of drift cases.
FR6 — Snowflake Lineage Overlay (Source-of-Truth Check)
Map extracted objects to Snowflake metadata:
objects exist?
views already compute the metric?
duplicated transformations across views?
Identify layer violations:
“Metric computed in SSRS query but should be in Snowflake semantic view”
Suggest candidate “gold views” (where to centralize metric).
Acceptance: lineage map resolves >95% of referenced objects for in-scope schemas.
FR7 — Rationalization Recommendations Engine
For each metric cluster and report:
Recommend one of:
Standardize (choose canonical definition)
Consolidate (merge reports / datasets)
Move to Snowflake (create curated semantic view)
Retire (low usage + redundant)
Rebuild (high drift/high risk)
Provide rationale:
Drift severity
Usage (if available)
Complexity score
Business criticality tag (manual override allowed)
Acceptance: recommendations are explainable and traceable to evidence.
FR8 — UI / Outputs (Executive-Ready)
Minimum deliverables:
Metric Drift Scorecard
#clusters, #drifted, top drifted KPIs
Duplicate Reports List
report pairs with similarity %
KPI Dictionary Export
definition, lineage, owner fields
Action Backlog Export
Excel/CSV/Jira-ready
UI options:
Web dashboard (preferred)
Or a generated PowerPoint + Excel pack
Acceptance: one-click export pack for stakeholders.
